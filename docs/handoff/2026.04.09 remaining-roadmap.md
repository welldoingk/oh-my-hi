### #1 터미널 가시성 히트맵 (Medium)

**목적**: Context Explorer의 3-state eye 데이터(`vis: 'hidden' | 'brief' | 'full'`)를 skill/agent 단위로 전역 집계해서 "어떤 스킬이 hidden 출력을 많이 생성하는가" 인사이트를 제공.

**데이터 소스**:
- 이미 `buildSessionEvents()`가 각 이벤트에 `vis` + `contextName`을 붙인다 → 별도 파싱 불필요
- 단, 현재는 세션 단위로만 사용. 전역 집계는 `listReplayableSessions()`를 iterate 하거나 `tokenEntries`의 `context` + `contextName` 필드로 직접 집계 가능

**필요 작업**:
1. `templates/session-events.mjs`에 `aggregateVisibilityByContext(usage)` 함수 추가
   - 반환 shape: `Map<contextName, { hidden: count, brief: count, full: count, totalTokens: number }>`
   - 순수 함수 (usage → 집계 결과) — 기존 모듈 패턴 유지
2. Overview 페이지 또는 Structure 페이지에 히트맵 섹션 추가
   - 추천: `#overview` 하단에 "Terminal Output Visibility" 카드
   - 렌더 옵션: (a) custom SVG grid (skill × vis 3-column heatmap), (b) billboard.js stacked bar
3. i18n 키 5-7개 (en/ko):
   - `visHeatmapTitle`, `visHeatmapDesc`
   - `visHidden`, `visBrief`, `visFull` (기존 `cwe_visHidden` 재사용 고려)
   - `visHeatmapInsightHidden` ("대부분 hidden" 인사이트 문구)
4. 단위 테스트 (`test/session-events.test.mjs`에 추가):
   - 빈 usage → 빈 Map
   - 단일 skill의 hidden/brief/full 분포 집계
   - totalTokens 합산 검증
5. UI 구조 테스트 (`test/web-ui.test.mjs`): 함수 존재, locale 키 존재

**난이도**: 🟡 중간. 데이터는 이미 파싱되어 있으므로 집계 + 렌더만 추가.

**참고**:
- `templates/session-events.mjs:83` 의 `buildSessionEvents`에서 `vis` 할당 로직 참조 (user prompt → 'full', skill/mcp → 'brief', hook → 'hidden')
- Context Explorer 범례 아이콘 SVG는 `templates/app.js:3822` 근처에 있음 — 재사용 가능

### #2 대시보드 초기 로드 lazy-load (Performance, High Complexity)

**문제**: `data.js`가 2–5MB 인라인 → 초기 HTML 파싱 시 블로킹. 사용자 체감: 첫 로드가 느리고 스피너 없이 멈춰 보임.

**현재 구조**:
- `scripts/generate-dashboard.mjs` → `writeDataJs()` → `data.js` + `data.json`
- `dashboard.html:42`: `<script src="data.js">` — 전체 데이터를 한 번에 로드
- `app.js`: 전역 변수 `DATA`로 즉시 접근

**옵션 비교**:

| 전략 | 장점 | 단점 |
|---|---|---|
| **카테고리별 분할** (`data-skills.js`, `data-agents.js`...) | 초기 로드 줄임 | HTTP 요청 수 증가, 단일 파일 배포 철학 깨짐 |
| **JSON 스트리밍** (`data.json` + fetch) | 단일 파일 유지 | file:// 환경에서 fetch 불가 (CORS) → 사용 불가 |
| **압축** (gzip/brotli string) | 파일 크기 감소 | decompressor 필요, 로드 시 파싱 → 여전히 블로킹 |
| **Progressive data loading** (우선 렌더 후 lazy merge) | 체감 성능 개선 | 일관성 이슈 (부분 데이터로 렌더) |
| **IndexedDB 캐시** | 재방문 시 빠름 | 첫 로드는 동일, file:// 환경 제약 |
| **web worker로 파싱** | UI 블로킹 해제 | 복잡도 급증 |

**추천 접근**: **progressive data loading**
- `data.js`를 **필수 부분** (scopes, categoryCounts, 최근 7일 usage)과 **deferred 부분** (전체 usage, transcripts)로 분할
- 필수 데이터로 Overview/Tokens 페이지 즉시 렌더
- `requestIdleCallback` 또는 script defer 로 전체 데이터 병합
- 카테고리 페이지 접근 시 "loading..." 스피너 표시 + 병합 완료 후 렌더

**필요 작업**:
1. 빌드 시 `writeDataJs` 를 `writeDataJsCore` + `writeDataJsFull` 로 분리
2. `dashboard.html` 에 `<script src="data-core.js">` + 동적 로드 `data-full.js`
3. `app.js` 에 `dataLoadState` 상태 (`'core' | 'full'`) + 페이지별 대기 로직
4. 테스트: core만 있어도 Overview 렌더 성공, full 병합 후 Category 렌더 성공
5. 마이그레이션 지원 (기존 data.js 폴백)

**난이도**: 🔴 높음. `DATA` 전역 접근 패턴이 파일 전반에 퍼져 있어서 getter 추상화가 필요할 수 있음.

**전제 질문** (작업 전 확인):
- 실제 2–5MB 로드가 얼마나 느린가? 우선 측정 필요 (Chrome DevTools Performance tab)
- 사용자 피해가 명확한가? (측정 없이 최적화 금지 원칙)

### #3 Skill/Agent 실행 트레이스 그래프 (New Feature, High Complexity)

**목적**: 현재는 skill/agent 호출 **횟수**만 표시. 호출 **체인**(A → B → C)을 시각화하면 복잡한 워크플로우의 의존관계 파악이 가능.

**데이터 모델 변경 필요**:

현재 `tokenEntries` 는 개별 API 턴만 기록 → 호출자/피호출자 관계가 없음.

가능한 소스:
1. **Transcript 파싱 확장**: `~/.claude/projects/*/sessions/*/*.jsonl` 에서 `tool_use`와 `tool_result`의 `parent_tool_use_id` (있다면) 필드로 tree 구성
2. **Heuristic**: 연속된 동일 세션에서 skill X 호출 직후 N초 내 skill Y 호출 → 부모-자식 관계로 추정
3. **Task tool 특수 처리**: `Task({subagent_type: X})` 호출 시점을 agent invocation 으로 간주하고 그 안의 도구 사용을 자식으로 묶기

**추천 방향**: 옵션 1 (정확한 소스). 단, 해당 필드가 없다면 옵션 3 (Task tool에 대해서만 부분 구현) fallback.

**필요 작업**:
1. `scripts/parsers/usage.mjs` 에서 transcript 파싱 시 parent_tool_use_id 수집 (필드 존재 확인 필요)
2. 데이터 shape 추가: `callChains: Array<{root: string, children: Array<{name, tokens, callCount}>}>`
3. 신규 페이지 `#execution-graph` 또는 Structure 페이지 확장
4. 렌더링: Mermaid/D3 force-directed graph, 또는 sankey (billboard.js 지원 안 함 → custom SVG 또는 별도 lib)
5. 대용량 체인은 top-N filter 제공

**난이도**: 🔴 높음. 데이터 모델부터 파서, 렌더러까지 전반적 변경.

**선결 조건**:
- transcript JSON 파일에 `parent_tool_use_id` (또는 유사 필드)가 실제로 존재하는지 확인
- 없으면 작업 scope 축소 (Task tool 한정 또는 포기)

### #4 Per-invocation 성능 프로파일링 (New Feature, High Complexity)

**목적**: 현재는 응답 지연 전체 평균만 표시. Skill별 init / execution / I/O 단계 분해로 병목 식별.

**데이터 수집 필요**:

`tokenEntries`의 `requestStartTime`/`requestEndTime`만으로는 step 단위 분해 불가. 필요한 추가 데이터:
- Tool use 시작 시점 (`tool_use_start`)
- Tool use 종료 시점 (`tool_use_end`)
- API 응답 수신 시점

**실현 가능성**:
- Claude Code transcript에 도구 호출 timing이 기록되는지 확인 필요
- 기록되지 않으면 **전체 구현 불가** → 이 아이템은 드롭하거나 Claude Code 자체 기능 요청으로 전환

**권고**: 본격 착수 전 transcript JSON 샘플을 열어보고 `timestamp` 또는 `duration_ms` 필드가 tool_use/tool_result에 있는지 확인. 없으면 이 아이템을 **handoff에서 제거**하고 "Claude Code 본체 요청" 으로 분리.

**난이도**: 🔴 매우 높음 (데이터 가용성에 종속).
